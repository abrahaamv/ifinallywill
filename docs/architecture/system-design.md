# System Architecture

## ğŸ¯ Architecture Philosophy

**Principles**:
1. **Mode-based optimization** - Different architecture for different use cases
2. **Cost-conscious design** - Start with WebSocket chat, upgrade to LiveKit for media
3. **Multi-tenant by default** - Isolation at every layer with RLS enforcement
4. **Horizontal scalability** - Stateless services, sticky sessions for WebSocket
5. **Progressive enhancement** - Basic features work everywhere, advanced features when supported

> **ğŸš¨ SECURITY FIRST**: All development must start with security patching:
> - Redis 7.4.2+ (4 RCE vulnerabilities)
> - PostgreSQL 17.3/16.7/15.11+ (SQL injection actively exploited)
> - Fastify 5.3.2+ (content-type parsing bypass)
>
> **ğŸ’° BUDGET REQUIREMENT**: LiveKit Enterprise plan ($5K-10K+/month) required for production

**Generic Code Name**: `platform` (used throughout codebase)

## ğŸ¨ Multi-App Architecture (Phase 1+)

**Architectural Decision**: Separate apps for different domains and user types.

### App Breakdown

| App | Domain | Purpose | Users | Port (Dev) |
|-----|--------|---------|-------|------------|
| **landing** | www.platform.com | Public marketing, signup | Public | 5173 |
| **dashboard** | dashboard.platform.com | Knowledge/RAG management | Tenant admins | 5174 |
| **meeting** | meet.platform.com | Real-time meetings | Participants | 5175 |
| **widget-sdk** | customer-site.com | Embeddable widget | End users | 5176 |

**Key Decisions**:
1. **Landing in Monorepo**: Early stage, shared auth/API, design consistency via `packages/ui`
2. **Shared UI Package**: Button, Input, Modal, Card, Typography, Icons shared across all 4 apps
3. **Independent Deployment**: Each app can be deployed separately to different subdomains
